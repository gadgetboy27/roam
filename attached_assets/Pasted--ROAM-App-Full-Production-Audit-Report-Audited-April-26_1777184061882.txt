## 🔴 CRITICAL — Fix Before Any Public Traffic

### C1. Socket.IO Has No Authentication Whatsoever
**File:** `server/routes.ts` ~line 163  
**Problem:** Any anonymous WebSocket connection can join the `send_message` handler. The server trusts `data.senderId` from the client payload — there is no token or session verification on the socket connection itself. A user can claim to be any `senderId` and send messages into any matched conversation they can guess the `matchId` for.  
**Fix:** Add auth handshake middleware to Socket.IO:
```ts
io.use(async (socket, next) => {
  const token = socket.handshake.auth?.token;
  if (!token) return next(new Error("Unauthorized"));
  const { data: { user }, error } = await supabaseAdmin.auth.getUser(token);
  if (error || !user) return next(new Error("Unauthorized"));
  const dbUser = await storage.getUserByEmail(user.email!);
  if (!dbUser) return next(new Error("Unauthorized"));
  socket.data.userId = dbUser.id; // attach verified ID
  next();
});
```
Then replace `data.senderId` checks with `socket.data.userId` throughout.

### C2. Supabase RLS Policy is Fully Open — Anyone Can Read or Write Any Message
**File:** `supabase_setup.sql` lines 26–31  
**Problem:** The RLS policy for the `messages` table is:
```sql
CREATE POLICY "Allow all on messages" ON public.messages
  FOR ALL USING (true) WITH CHECK (true);
```
This means any authenticated Supabase user can read messages from any match, write messages with any `sender_id`, and update the `read` field on any record. The client-side `sendSupabaseMessage()` and `markMessagesRead()` functions write directly to Supabase — bypassing your Express API entirely.  
**Fix:** Replace the open policy with proper RLS that enforces participant membership:
```sql
-- Users can only see messages in matches they're part of
CREATE POLICY "messages_select" ON public.messages FOR SELECT
  USING (
    auth.uid()::text IN (
      SELECT user_a_id FROM matches WHERE id = match_id
      UNION
      SELECT user_b_id FROM matches WHERE id = match_id
    )
  );
-- Users can only insert messages as themselves
CREATE POLICY "messages_insert" ON public.messages FOR INSERT
  WITH CHECK (auth.uid()::text = sender_id);
-- Users can only mark read on their own received messages
CREATE POLICY "messages_update" ON public.messages FOR UPDATE
  USING (auth.uid()::text != sender_id AND match_id = match_id);
```
Apply the same principle to `typing_indicators`.

### C5. In-Memory Rate Limiter Bypassed on Every Server Restart
**File:** `server/routes.ts` lines 22–56  
**Problem:** The custom rate limiter uses `new Map()` stored in module scope. On Replit (and any serverless/restart-prone environment), every cold start resets all rate limit counters. Brute-force login attempts only need to trigger a restart to reset their count. This defeats login protection, signup throttling, and upload limiting.  
**Fix:** Use Redis-backed or database-backed rate limiting. On Replit/Postgres, use `express-rate-limit` with `rate-limit-postgresql` or the existing session Postgres pool:
```bash
npm install express-rate-limit @express-rate-limit/postgresql
```
Alternatively, use Upstash Redis (free tier) which is stateless-friendly.

### H3. Password Column Still Exists and is NOT NULL on Users Table
**File:** `shared/schema.ts` line 14, `migrations/*.sql`  
**Problem:** `users.password` is `text NOT NULL` — the legacy scrypt hash column is still the primary user record. New users created via Supabase OAuth or SSO still need a value in this column. The dual auth system means:
- Legacy users: session auth + scrypt hash in DB
- New users: Supabase JWT auth + password column contains a placeholder or random value
- The `migrate-to-supabase` endpoint silently re-registers users in Supabase

This creates an indefinite security surface: the legacy auth path (`/api/auth/login`) with scrypt + session tokens remains active, meaning users can bypass Supabase's MFA, lockout, and audit features.  
**Fix (migration path):**
1. Make `password` nullable in the schema
2. Add a `auth_provider` column: `enum('local', 'supabase')`
3. Set a hard sunset date for the legacy path
4. Disable `/api/auth/login` after migration is complete
5. Ensure `/api/auth/migrate-to-supabase` requires a valid existing session before it'll create a Supabase account

### H4. No Content-Security-Policy Header
**File:** `server/index.ts` lines 27–35  
**Problem:** The app sets `X-Content-Type-Options`, `X-Frame-Options`, and `X-XSS-Protection`, but there is no `Content-Security-Policy` header. Without CSP, any XSS vulnerability (e.g., in user-generated photo captions or taglines) can execute arbitrary scripts, steal sessions, or exfiltrate data.  
**Fix:** Install Helmet and configure CSP:
```bash
npm install helmet
```
```ts
import helmet from "helmet";
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      imgSrc: ["'self'", "data:", "https://znqbnldsalsfpraiplxz.supabase.co", "https://*.supabase.co"],
      connectSrc: ["'self'", "https://*.supabase.co", "wss://*.supabase.co"],
      scriptSrc: ["'self'"],
      styleSrc: ["'self'", "'unsafe-inline'"], // tighten after audit
    }
  },
  hsts: { maxAge: 31536000, includeSubDomains: true }
}));
```

| Table | Column | Reason |
|---|---|---|

### H6. N+1 Query Problem in `/api/discover` — Will Not Scale
**File:** `server/routes.ts` ~line 508  
**Problem:** The discover endpoint calls `storage.getPhotosByUser(candidate.id)` for each candidate user inside a `Promise.all()`. With 100 users this is 100 DB queries per page load. With 1,000 users it's 1,000 queries. Additionally `storage.getAllUsers()` loads every user with no pagination.  
**Fix:**
1. Replace the per-user photo fetch with a single bulk query:
```ts
const allPhotos = await storage.getFirstApprovedPhotoPerUser(); // already exists!
```
2. Add cursor-based pagination to `/api/discover` (page, limit params)
3. Cache the fingerprint computation in Redis or a DB column

### H7. No Foreign Key Constraints on Any Relationship
**File:** `migrations/0000_cold_mariko_yashida.sql`  
**Problem:** No `REFERENCES` constraints exist anywhere in the schema. This means:
- Photos can reference non-existent users
- Messages can reference non-existent matches
- Group members can reference deleted groups
- Notifications accumulate for deleted users

This causes silent data corruption and makes cleanup impossible.  
**Fix:** Add FK constraints in Drizzle schema using `.references()`:
```ts
userId: varchar("user_id").notNull().references(() => users.id, { onDelete: "cascade" }),
matchId: varchar("match_id").notNull().references(() => matches.id, { onDelete: "cascade" }),
```
Then regenerate migrations. Use `onDelete: "cascade"` for child records, `onDelete: "set null"` for optional references.

### H8. `uploads/` Directory Publicly Served via Express Static
**File:** `server/index.ts` line 37  
**Problem:** `app.use("/uploads", express.static(...))` serves the local `uploads/` folder. Although Supabase Storage is the primary upload path, this static mount means any file placed in the `uploads/` directory (e.g., by a path traversal exploit, misconfiguration, or future code change) becomes publicly accessible at a guessable URL.  
**Fix:** Remove this middleware entirely since all uploads go to Supabase Storage. If legacy local uploads must remain, serve them through a gated API endpoint that validates the requesting user has permission.

## 🔵 ARCHITECTURE & COST OBSERVATIONS

### A1. Dual Messaging System (Socket.IO + Supabase Realtime) — Pick One
The app uses Socket.IO for real-time delivery **and** Supabase Realtime (Postgres changes) for message subscriptions. This means every message goes through two real-time channels. Supabase Realtime on the free plan has connection limits. Consolidate on one system — Supabase Realtime is the better long-term choice as it removes the need for socket.io server infrastructure entirely.

### A2. Supabase `messages` Table is Separate from Drizzle `messages` Table
The `supabase_setup.sql` creates a `public.messages` table in Supabase directly. The Drizzle `messages` table in Postgres is a **different table** in your self-managed Postgres DB. You now have two messages tables storing duplicate/overlapping data. This is a data integrity nightmare. The architecture needs to be clarified: either use Supabase as the single DB (with Drizzle pointing at it via `DATABASE_URL=supabase-pooler-url`) or use your own Postgres for everything and drop the Supabase Realtime messaging.

### A3. `/api/discover` Loads All Users Into Memory on Every Request
No pagination exists. As user count grows this will exhaust memory and slow response times dramatically. Implement server-side pagination immediately.

### A4. Photo Upload Uses Base64 Data URLs (25MB JSON Limit)
Base64 encoding adds ~37% overhead. An 8MB photo becomes ~11MB in transit. Consider switching to signed Supabase Storage upload URLs (the client uploads directly to Supabase, bypassing your server) which is both faster and cheaper.

### A5. Stripe `boostExpiresAt` Not Indexed — Boost Feature Will Fail at Scale
The discover feed logic presumably filters boosted users by `boost_expires_at`. Without an index on this column the query scans all users. Add to the index migration (see H5).
